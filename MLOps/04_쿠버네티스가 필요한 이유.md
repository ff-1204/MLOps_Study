# MLOps에서 쿠버네티스가 필요한 이유

---

## 도커와 쿠버네티스의 역할 구분

| 도구 | 하는 일 | 비유 |
|---|---|---|
| **Docker (도커)** | 컨테이너 하나를 만들고 실행 | 컨테이너 박스 하나 만들기 |
| **Kubernetes (쿠버네티스)** | 수많은 컨테이너를 자동으로 관리·운영 | 수백 개 컨테이너 박스를 조율하는 항만 관리 시스템 |

> **컨테이너(Container)**: 코드 + 실행 환경(Python 버전, 라이브러리 등)을 하나의 박스에 담아 **어디서든 동일하게 실행**할 수 있게 만든 기술.  
> 도커는 Ubuntu, Python, TensorFlow, PyTorch 같은 환경을 통째로 패키징한다.

```
도커  = 컨테이너 하나를 만들고 실행하는 도구
쿠버네티스 = 수많은 컨테이너를 조율하고 운영하는 도구
```

---

## MLOps에서 쿠버네티스가 필요한 3가지 이유

### 1. Reproducibility (재현성) — 실행 환경의 일관성 & 독립성

**문제 상황**: A팀원은 Python 3.8, B팀원은 Python 3.10을 쓰고 있어서 같은 코드가 다르게 동작한다.

쿠버네티스는 컨테이너 단위로 실행 환경을 완전히 격리해서 **누가 어디서 실행해도 동일한 결과**를 보장한다.

- 학습 환경과 서빙 환경의 불일치 문제 해결
- 팀원 간 환경 차이로 인한 재현 불가 문제 해결

---

### 2. Job Scheduling (작업 스케줄링) — 스케줄 관리 · 병렬 작업 관리 · 유휴 자원 관리

**문제 상황**: GPU 서버가 1대인데 팀원 5명이 동시에 모델을 학습시키려 한다. 누가 먼저 써야 하나?

쿠버네티스 없이는 카카오톡이나 엑셀로 수동으로 순서를 정해야 한다.  
쿠버네티스가 있으면 이를 자동으로 처리한다.

| 기능 | 설명 |
|---|---|
| **스케줄 관리** | 학습 작업을 언제 어느 서버에서 실행할지 자동 결정 |
| **병렬 작업 관리** | 여러 실험을 동시에 실행하고 GPU·CPU를 나눠서 배분 |
| **유휴 자원 관리** | 놀고 있는 GPU를 다른 팀의 작업에 자동으로 할당 |

---

### 3. Auto-healing & Auto-scaling — 장애 대응 · 트래픽 대응

**문제 상황**: 밤 12시에 서빙 서버가 갑자기 죽었다. 누가 깨어나서 재시작해야 하나?

| 상황 | 쿠버네티스 자동 대응 |
|---|---|
| 서빙 중 컨테이너가 죽으면? | **Auto-healing**: 자동으로 재시작 — 사람이 몰라도 됨 |
| 트래픽이 갑자기 몰리면? | **Auto-scaling**: 서버(Pod)를 자동으로 늘림 |
| 트래픽이 줄어들면? | 서버를 자동으로 줄여서 비용 절감 |

> **Pod(파드)**: 쿠버네티스에서 컨테이너를 실행하는 최소 단위. 쉽게 말해 "컨테이너를 담는 작은 실행 공간"이라고 이해하면 된다.

---

## 쿠버네티스의 핵심 원리 — 선언형 인터페이스

쿠버네티스는 **"어떻게 해줘"가 아니라 "이런 상태가 됐으면 좋겠어"** 방식으로 동작한다.

| 방식 | 예시 |
|---|---|
| **명령형** (How) | "A 컨테이너를 3번 재시작하고, 포트를 8080으로 열고, 메모리를 2GB 할당해줘" |
| **선언형** (What) | "A 컨테이너가 3개 실행되고 있었으면 좋겠어" (Desired State) |

```
에어컨 비유:
- 명령형: 냉매를 어떤 걸 쓰고, 얼마나 압축한 뒤, 어떻게 순환시켜서...
- 선언형: 내 방 온도가 20도가 되었으면 좋겠어  ← 쿠버네티스 방식
```

쿠버네티스는 선언한 상태(Desired State)를 항상 유지하려 한다.  
Pod가 1개 죽으면 → 자동으로 1개 새로 실행. 트래픽이 늘면 → 자동으로 Pod 추가.

## 쿠버네티스 클러스터 구조

쿠버네티스는 **Control Plane(두뇌)** 과 **Worker Node(일꾼)** 로 구성된다.

```
Kubernetes Cluster (전체 관리 단위)
│
├── Control Plane (두뇌 — 명령을 내리는 곳)
│   ├── API Server       : 모든 명령이 들어오는 창구
│   ├── Scheduler        : "이 작업은 저 서버에서 실행해"를 결정
│   ├── Controller Manager: 선언한 상태(Desired State)가 유지되는지 계속 감시
│   └── etcd             : 클러스터의 모든 상태 정보를 저장하는 저장소
│
└── Worker Node (일꾼 — 실제 컨테이너가 실행되는 서버들)
    ├── kubelet          : Control Plane의 명령을 받아 Pod를 실행·관리
    └── kube-proxy       : 네트워크 트래픽을 올바른 Pod로 연결
```

## 로컬에서 쿠버네티스 실습하기

실제 서버 클러스터 없이도 내 컴퓨터에서 쿠버네티스를 연습할 수 있다.

| 도구 | 특징 | 추천 대상 |
|---|---|---|
| **minikube** | 가장 많이 쓰이는 로컬 실습 도구 | 처음 배우는 사람 |
| **k3s** | 가볍고 빠름 (100MB 이하) | 라즈베리파이 등 소형 기기 |
| **MicroK8s** | Ubuntu에서 명령어 한 줄로 설치 | Ubuntu 사용자 |

## 퍼블릭 클라우드의 쿠버네티스 서비스

직접 클러스터를 구축하지 않고 클라우드에서 바로 쓸 수도 있다.

| 벤더 | 서비스명 |
|---|---|
| AWS | Amazon EKS (Elastic Kubernetes Service) |
| GCP | Google Kubernetes Engine (GKE) |
| Azure | Azure Kubernetes Service (AKS) |

---

## Container Orchestration 시장 점유율

CNCF 조사(2018) 기준 쿠버네티스 점유율은 **83%** 로 압도적 1위.  
사실상 업계 표준이 되어 MLOps 도구들(Kubeflow, Argo Workflows 등)이 모두 쿠버네티스 위에서 동작한다.
